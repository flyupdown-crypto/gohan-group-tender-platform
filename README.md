# Gohan Group 集团数字化系统采购与供应商评估平台

这是一个纯静态、无后端、无数据库、可直接浏览器打开、可部署到 Vercel 的企业级 SaaS 招标与供应商评估平台。

平台定位：

**Gohan Group 跨业态数字化系统招标与供应商评估平台**

## 1. 项目背景

Gohan Group 是跨业态集团，业务覆盖：

- 餐饮
- 零售
- 未来酒店
- 娱乐
- 连锁门店

目前集团已有多个独立系统，包括餐饮系统、零售 POS、会员系统、供应链系统和财务系统。现有供应商存在明显局限：某些供应商仅懂餐饮，某些供应商仅懂零售，没有供应商天然具备完整跨业态整合能力。

本次招标重点不是单店 POS，而是：

- 集团级数字化
- ERP / WMS 深度能力
- 跨业态会员体系
- 多系统数据整合
- 集团级数据中台
- 长期扩展能力

重要原则：任何供应商都必须存在一定程度定制开发，不得使用低价标准功能误导客户。

## 2. 编号规则

模块编号采用：

- `M01`
- `M02`
- `M03`

需求编号采用：

- `M01-R001`
- `M01-R002`

编号在供应商视图、老板视图、风险列表、JSON 导出、CSV 导出中保持一致。

## 3. 需求模块

当前已从 245 个需求项瘦身为 8 个一级模块、110 个细分需求项：

- M01 项目背景与供应商资格：8 项
- M02 跨业态会员与客户数据整合：20 项
- M03 ERP / WMS 深度能力：30 项
- M04 供应链与采购库存管理：3 项
- M05 多系统集成与硬件适配：25 项
- M06 数据分析与老板决策：3 项
- M07 实施、迁移与上线交付：13 项
- M08 报价、隐性成本与风险披露：8 项

瘦身原则：删除基础、通用、低差异化功能；保留 ERP / WMS、跨业态、实施迁移、架构接口、GDPR、欧洲部署、硬件适配和隐性成本披露等能拉开供应商差距的硬核能力。

## 4. 页面结构

当前版本已升级为可演示的企业级 WebApp，包含四个核心页面：

- 首页总览：展示 GOHAN GROUP 品牌、项目背景、三步使用流程和三类角色入口。
- 老板决策视图：用于客户老板查看总评分排名、C/D/E/F 能力图表、高风险需求和低价高风险提示。
- 供应商填写视图：用于供应商逐项填写支持情况、实现方式、工时费用、第三方依赖、证据与风险，并显示填写进度和必填缺失提示。
- 管理员评估视图：用于查看全部供应商数据、需求结构统计、导出按钮和风险规则说明。

三步使用流程：

1. 客户确认项目背景与需求范围。
2. 供应商逐项填写功能响应、报价与实施风险。
3. 客户查看供应商横向对比与风险评分。

## 5. 访问模式与前端隔离

当前版本支持独立入口页面，适合部署后分别转发给客户老板、供应商和管理员：

- 老板视图：`owner.html`
- 供应商填写视图：`supplier.html`
- 管理员评估视图：`admin.html`

部署到 Vercel 后，推荐使用以下链接：

- 老板视图：`https://你的项目.vercel.app/owner.html`
- 供应商填写视图：`https://你的项目.vercel.app/supplier.html`
- 管理员评估视图：`https://你的项目.vercel.app/admin.html`

微信转发给外部供应商时，建议只发送：

```text
https://你的项目.vercel.app/supplier.html
```

系统仍兼容通过 URL 参数进入不同演示模式：

- 老板视图：`index.html?mode=owner`
- 供应商填写视图：`index.html?mode=supplier`
- 管理员评估视图：`index.html?mode=admin`

如果没有 `mode` 参数，默认进入 `owner` 模式。

供应商模式只渲染：

- GOHAN GROUP 项目说明简化版
- 供应商资料填写
- 供应商自声明
- 需求填写卡片
- 填写进度
- 保存状态
- 提交 / 导出自身填写数据按钮

供应商模式不会渲染老板视图、管理员视图、总评分排名、供应商横向对比、其他供应商数据、权重分数、评分公式、风险排名、老板决策图表或管理员全量导出按钮。

供应商填写页不再显示 `供应商 A / B / C` 编号。供应商只需填写公司名称、联系人、邮箱、官网和报价响应。导出的 JSON 文件以公司名称命名，便于管理员收集。

静态版本的数据收集方式：

1. 供应商打开 `supplier.html` 填写。
2. 供应商点击“提交 / 导出自身填写数据”下载 JSON。
3. 供应商将 JSON 发给管理员。
4. 管理员在 `admin.html` 导入一个或多个供应商 JSON。
5. 老板视图和管理员视图按公司名称展示并横向比较所有已导入供应商。

因此，静态版本不再依赖固定的 `供应商 A / B / C` 入口；只要供应商提交了 JSON，就可以进入对比。

安全提醒：

当前版本通过独立入口页面和 URL `mode` 参数实现前端视图隔离，仅用于 Demo 和招标流程演示；正式生产环境应增加真实登录、权限校验和后端数据隔离。

## 6. 能力分类

所有需求都归入以下能力分类：

- A 类：通用基础功能
- B 类：企业级运营管理功能
- C 类：ERP / WMS 深度能力
- D 类：跨业态 / 跨模块能力
- E 类：实施与迁移能力
- F 类：架构与扩展能力
- H 类：硬件与设备能力

## 7. 权重逻辑

老板视图显示完整权重，供应商填写视图不显示权重。

权重规则：

- A 类权重较低
- B 类权重中等
- C / D / E / F 类权重较高
- H 类用于单独识别硬件适配风险
- ERP 深度能力、跨业态能力、实施迁移能力、架构扩展能力和硬件适配风险是重点评估对象

每个需求包含：

- `weightLevel`
- `weightScore`

## 8. 供应商填写字段

供应商填写页包含默认预填值，用于降低首次填写成本。默认值为：

- 是否支持：支持
- 功能深度：标准支持
- 是否标准功能：标准功能
- 预计开发工时：0 人天
- 费用：0
- 风险说明：空白

这些默认值仅用于初始预填，供应商可以自行修改。正式提交前供应商应逐项确认，默认值不代表平台承诺；未修改内容将被视为供应商确认。若浏览器已有 LocalStorage 保存内容，系统会优先读取已保存内容，不会用默认值覆盖用户已填写数据。

每个需求项供应商必须填写：

- 是否支持
- 功能深度
- 是否标准功能
- 预计开发工时（人天）
- 费用
- 风险说明

以下内容改为供应商级别统一填写一次，不再在每个需求项中重复填写：

- 是否支持 GDPR
- 是否已在欧洲部署
- 是否具备 Open API
- 是否依赖第三方系统
- 第三方系统整体说明
- 是否提供案例

供应商填写页新增“项目总报价”区域：

- 软件总报价
- 实施服务费用
- 数据迁移费用
- 培训费用
- 硬件费用
- 第三方费用
- 年服务费 / SaaS 年费
- 需求项预计开发工时汇总
- 需求项费用汇总
- 项目总金额自动汇总
- 报价说明

金额统一使用欧元符号 `€`。项目总金额会自动汇总项目报价字段，并叠加所有需求项中的费用汇总。

## 9. 强制校验与风险机制

系统会自动识别：

- 选择“标准功能”但填写大量工时：提示“存在疑似隐性定制开发”
- 定制开发但未填写工时
- 定制开发但未填写费用
- 需求项费用未填写
- 供应商级第三方依赖说明不完整
- 项目总报价结构不完整

老板风险提示：

如果供应商基础功能得分高，但 ERP / WMS 深度、跨业态整合或实施迁移能力不足，系统会提示：

“该供应商基础功能覆盖较好，但 ERP/WMS 深度、跨业态整合或实施迁移能力不足，不建议仅依据基础功能覆盖率做决策。”

低价高风险提示：

“该供应商存在低价高风险特征：报价可能未完整覆盖实施、迁移、定制开发、接口或后续运维成本，建议进一步澄清。”

## 10. 老板视图说明

老板视图用于决策，不是供应商填表页面。它包含：

- 决策摘要卡片
- 总评分排名
- 推荐 / 谨慎 / 不建议标签
- 功能覆盖率
- 标准功能比例
- 定制开发比例
- 供应商报价对比
- 最低报价 / 最高报价 / 平均报价
- 高风险需求数量
- 隐性成本风险
- ERP 深度能力评分
- 跨业态能力评分
- 实施迁移能力评分
- 技术架构成熟度
- 硬件适配风险排名
- 风险热力图
- 权重分析
- 工时分布图
- 成本结构图
- 模块完成度图
- 高风险需求列表

老板视图和管理员视图只显示公司名称，不显示供应商编号。供应商报价对比不限制供应商数量，管理员可通过导入供应商 JSON 增加对比对象。

当前 Phase 2 不包含 PDF 导出。系统保留 JSON / CSV 导出；如需 PDF，建议后续增加打印友好的报告页，再由浏览器保存为 PDF。

## 12. Logo 替换说明

页面支持真实 Logo 文件，变量为 `logoPath`，默认路径：

```text
assets/logo.png
```

支持 PNG / SVG。若文件不存在，界面会回退到默认图标。

如需替换客户 Logo，请将 PNG 或 SVG 文件命名为 `logo.png`，并放入 `assets` 文件夹。

## 14. Vercel 部署说明

本项目为纯静态网页：

- 无外部依赖
- 无后端
- 无数据库
- 可直接打开 `index.html`
- 可直接部署到 Vercel

部署步骤：

1. 上传 `index.html`、`supplier.html`、`owner.html`、`admin.html`、`styles.css`、`app.js`、`README.md`、`.gitignore` 和 `assets` 文件夹到 GitHub。
2. 在 Vercel 选择从 GitHub 导入。
3. 构建命令留空。
4. 部署后使用 `.vercel.app` 链接演示。

当前阶段建议先使用 Vercel 自动生成的 `.vercel.app` 链接进行演示和验证，暂不绑定 FiveVectors 自有域名。

建议同时保留 `.gitignore`，避免 `.DS_Store`、`.vercel` 等本地临时文件上传到 GitHub。

部署完成后建议检查：

- `https://你的项目.vercel.app/supplier.html`
- `https://你的项目.vercel.app/owner.html`
- `https://你的项目.vercel.app/admin.html`
